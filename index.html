Current Architecture — The Problem

The existing BatchTransfer table stores all aggregation levels (Mono → Outer → Master → Pallet) in a single flat row per MonoCode. This leads to severe data duplication, compounded further when ShipperDetail copies the same data.

⚠
Core Issue: One PalletCode is repeated across 9,000 rows in BatchTransfer. On LineClearance, all those rows are bulk-copied into ShipperDetail — doubling the data with zero new information. As production volume grows, this pattern becomes unsustainable.

Proposed Schema — Hierarchical Design

Each aggregation level gets its own table. References use the actual code values (natural keys) instead of auto-increment IDs — this is critical because LineClearance deletes and re-inserts C_ table records, causing IDs to change. Codes remain stable.

✓
Why code-based FKs and not ID-based? During LineClearance, C_OuterCode / C_MasterCode / C_PalletCode records are deleted and fresh copies are inserted from Hive. This changes their auto-increment IDs. If BT_ tables reference by ID, those references become orphans. Codes (OuterCode, MasterCode, PalletCode) never change — they are the stable natural keys.

Full Hierarchy Query (via JOINs)
SELECT s.id AS ShipperID, s.BatchName, p.PalletCode, m.MasterCode, o.OuterCode, mo.MonoCode, mo.MonoCode_Result, mo.Grade FROM Shipper s JOIN ShipperDetail sd ON sd.ShipperID = s.id JOIN BT_Pallet p ON p.PalletCode = sd.PalletCode JOIN BT_Master m ON m.PalletCode = p.PalletCode JOIN BT_Outer o ON o.MasterCode = m.MasterCode JOIN BT_Mono mo ON mo.OuterCode = o.OuterCode WHERE s.id = @ShipperID
◈
Index these columns for fast JOIN performance: BT_Mono(OuterCode, BatchID) · BT_Outer(MasterCode, BatchID) · BT_Master(PalletCode, BatchID) · BT_Pallet(PalletCode)

📋
For Stakeholders: The current system copies 9,000 rows of data just to track which shipper a pallet belongs to. As production volume grows, the database grows at 2× the rate it should. The proposed change cuts that growth in half and eliminates data inconsistencies.
👨‍💻
For Developers: Queries become JOINs instead of flat SELECTs — a small complexity cost. In return: no stale copies, no orphan references after LineClearance, and clean separation of concerns per aggregation level. Adding indexes on code columns keeps performance fast.

Migration Plan

A phased approach to move from the existing flat BatchTransfer to the new hierarchical schema without disrupting production.

◈
Strategy: Create new BT_ tables alongside existing tables. Migrate historical data. Switch application writes to new tables. Validate. Decommission old tables. No downtime required if done in phases.
01
Create new BT_ tables
Create BT_Pallet, BT_Master, BT_Outer, BT_Mono alongside existing tables. Add indexes on code columns immediately. No existing data is touched.
02
Migrate historical BatchTransfer data
Run migration scripts in order: Pallet → Master → Outer → Mono. Each level uses DISTINCT to deduplicate. Run during off-peak hours. Validate row counts after each step.
-- Step 1: Pallet level (DISTINCT deduplication) INSERT INTO BT_Pallet (BatchID, PalletCode, PalletCode_Result, CreatedDate) SELECT DISTINCT BatchID, PalletCode, PalletCode_Result, MIN(CreatedDate) FROM BatchTransfer WHERE PalletCode IS NOT NULL GROUP BY BatchID, PalletCode, PalletCode_Result; -- Step 2: Master level INSERT INTO BT_Master (BatchID, MasterCode, MasterCode_Result, PalletCode, CreatedDate) SELECT DISTINCT BatchID, MasterCode, MasterCode_Result, PalletCode, MIN(CreatedDate) FROM BatchTransfer WHERE MasterCode IS NOT NULL GROUP BY BatchID, MasterCode, MasterCode_Result, PalletCode; -- Step 3 & 4: Same pattern for BT_Outer and BT_Mono
03
Update ShipperDetail
Create the new lean ShipperDetail (id, ShipperID, PalletCode, AddedDate). Migrate existing shipments by mapping batchtransferID → PalletCode. Old ShipperDetail kept as backup initially.
04
Switch application writes
Update the production line Hub application to write new aggregation results into BT_ tables instead of BatchTransfer. Update LineClearance process to write ShipperDetail as a single PalletCode reference.
05
Parallel run & validation
Run both old and new write paths simultaneously for 1–2 batches. Compare row counts and query results between old and new tables. Confirm full-hierarchy JOINs return correct data.
06
Decommission old tables
Once validation passes, stop writes to BatchTransfer and old ShipperDetail. Archive old tables with a date suffix (BatchTransfer_archive_YYYYMMDD) before dropping. Remove after 30-day retention period.
